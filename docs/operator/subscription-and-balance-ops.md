# 订阅与余额 — 运营速查

面向运营/客服：解释用户权益、后台配置、计费规则与客户常见问题。

---

## 1. 两个概念（先分清）

| | **订阅** | **余额（充值）** |
|---|---------|-----------------|
| 分组类型 | 订阅计费型 | 标准型（含专属分组） |
| 用户付平台的钱 | 购买订阅套餐（或管理员赠送） | 充值进账户 |
| 用量怎么扣 | 扣套餐 **日/周/月限额**（日=24h、周=7 天、月=30 天滚动窗口，需分组已配置对应限额） | 扣 **账户余额**（按量 × 分组倍率） |
| 同一笔请求 | 只走其中一种，**不会两边都扣** | 同左 |

**用户交给平台的真钱** = 充值金额 + 订阅购买金额。管理员分配、兑换码开通的不算用户付费。

---

## 2. 计费规则（客服必知）

- **按 Key 生效分组决定**（直连 Key 看绑定分组；全能 Key 每请求解析后端组），不是「余额/订阅/专属」三条线自动合并扣费。
- **直连 Key**（`routing_mode=direct`）：绑订阅组 → 订阅计费；绑标准组（公开或专属）→ 扣余额。
- **全能 Key**（新建 Key 默认）：每请求按模型+端点选后端组；同平台若既有订阅又有标准/专属组，**优先订阅组**（扣订阅限额，不动余额）。
- 订阅限额用尽 → 429，**不会自动改扣余额**。
- 订阅过期 → 403；若还有余额且 Key 绑标准/专属组，仍可继续用。
- Key 上的「额度限制」「速率限制（5h/1d/7d）」两种计费模式都生效，是 Key 级额外总闸。

## 3. 后台配置

### 3.1 建订阅套餐（分组）

**管理后台 → 分组管理** → 新建/编辑：

- 类型选 **「订阅计费」**
- 配置平台、倍率、日/周/月限额（USD）
- 状态保持 **启用**

### 3.2 给用户开通订阅

**管理后台 → 订阅管理 → 分配订阅**：

- 选用户、订阅分组、有效期（天，默认 30）
- 已有同组订阅要 **续期** → 用「调整订阅」，不要重复分配（否则可能冲突）

其他开通方式：用户 **购买页付款**、**订阅兑换码**、新用户 **默认订阅**（仅注册时）。

### 3.3 专属分组 + 余额

- **专属分组**：标准型 + 专属；在用户 **允许分组** 里授权。
- **充值**：用户购买页充值，或后台兑换码/人工充值流程；余额仅在 **标准型分组**（公开或专属）下消耗，订阅型 Key 不扣余额。

### 3.4 常用运维操作

| 操作 | 路径 |
|------|------|
| 延长/缩短有效期 | 订阅管理 → 调整订阅 |
| 用量清零 | 订阅管理 → 重置配额 |
| 停用 | 撤销订阅（状态变为已撤销）；恢复用「恢复订阅」。后台无单独「暂停」按钮 |

## 4. 用户使用（告知客户）

1. 开通订阅后，在 **API Keys** 创建 Key，**绑定对应订阅分组**（或用全能 Key，同平台优先走订阅）。
2. **「我的订阅」** 可查看到期时间、日/周/月用量及重置倒计时。
3. 订阅 **到期需主动续费**（购买页/续费按钮）；**不支持到期自动扣款**。到期提醒邮件依赖后台开关 `subscription_expiry_notify_enabled`（默认开启）。
4. Key 上的「额度限制」是 **累计上限**，不会自动清零；需手动重置。Key「速率限制（5h/1d/7d）」是滚动窗口，窗口到期会自动归零。套餐 **月限额** 按 30 天滚动自动重置（日/周限额同理按 24h/7 天滚动）。

---

## 5. 客户 FAQ（标准答复）

**Q：Key 能每月自动重置额度吗？**  
A：Key 自身累计额度不会自动重置。若指 **订阅套餐月限额**，已支持 30 天滚动自动重置。

**Q：订阅到期能自动扣款续费吗？**  
A：目前不支持；若后台开启到期提醒，会发邮件，但仍需用户主动下单续费。

**Q：订阅能按月额度重置吗？**  
A：可以。分组配置月限额后，用量每 30 天滚动归零；与订阅 **有效期** 是两回事，到期仍须续费。若套餐有效期 ≤1 天，日限额按「一次性」处理，随订阅到期结束，不按 24h 滚动重置。

**Q：既有订阅又充值，钱怎么算？**  
A：订阅费买套餐额度；充值供标准/专属分组按量使用。每次请求只扣一种，不会重复扣。

## 6. 场景速查

| 用户情况 | Key | 实际扣费 |
|---------|-----|---------|
| 有订阅 + 有余额 | 绑订阅组 | 订阅限额 |
| 有订阅 + 有余额 | 绑专属组 | 余额 |
| 有订阅 + 有余额 | 全能 Key（同平台两都有） | 优先订阅限额 |
| 订阅限额用尽 | 绑订阅组 | 拒绝（不扣余额） |
| 订阅过期 | 绑订阅组 | 拒绝 |
| 订阅过期 | 绑专属组 + 有余额 | 扣余额 |

## 7. 配置检查清单

给用户配完权益后确认：

- [ ] 订阅状态 **生效中**、未过期  
- [ ] 分组日/周/月限额符合预期  
- [ ] 用户 Key **绑对分组**（或确认全能 Key 路由符合预期）  
- [ ] 用户能在「我的订阅」看到用量与到期时间

## 8. 补充：平台限额 & 续期方式

### 8.1 用户 × 平台限额（与订阅限额是两层）

**是什么**：按 **用户 + 平台**（anthropic / openai / gemini 等）配置的日/周/月消费上限（USD），滚动窗口与订阅相同（24h / 7 天 / 30 天）。

**何时生效**：仅在 **余额计费**（标准型分组 Key）下校验并累加用量。**订阅型 Key 不受此限**——走订阅分组自己的日/周/月限额。

**与订阅限额的关系**：并行存在、互不替代。例如用户有 Claude 订阅又有余额：绑订阅组 → 只看订阅限额；绑标准 Claude 组 → 看余额 + 平台限额（若已配置）。

**后台怎么配**：

| 配置项 | 路径 | 说明 |
|--------|------|------|
| 新用户默认 | **设置 → 用户 → 默认平台限额** | 仅 **注册时** 写入；已有用户不自动改 |
| 单用户覆盖 | **用户管理 → 某用户 → 平台限额** | 可逐平台改日/周/月；留空=继承默认；填 **0**=禁止该窗口 |
| 用户自查 | 用户侧接口/页面（平台限额） | 看各平台用量与重置时间 |

**客服话术**：「平台限额管的是 **充值按量** 那条线；套餐订阅限额管 **订阅 Key** 那条线，两套分开算。」

### 8.2 续期 / 延长期：支付 vs 后台

三种入口，**都是改 `expires_at`**，但适用场景不同：

| 方式 | 入口 | 何时用 | 到期日怎么算 |
|------|------|--------|-------------|
| **用户支付续费** | 购买页 / 「我的订阅」→ 续费 | 用户自助付费 | **未过期**：在原到期日上 **+ 购买天数**；**已过期**：从 **当天** + 购买天数并激活 |
| **后台调整订阅** | 订阅管理 → 调整订阅 | 赠送天数、补偿、人工缩短 | **未过期**：在原到期日上 **± 调整天数**；**已过期**：仅允许 **正数**，从当天加天数并激活 |
| **后台分配订阅** | 订阅管理 → 分配订阅 | 用户 **尚无** 该分组订阅 | 从分配日起 + 有效期（默认 30 天） |

**注意**：

- 用户 **已有同组订阅** 要加时，优先引导 **支付续费** 或后台 **调整**；不要重复「分配」（参数不一致会报冲突）。
- **支付续费** 会生成订单、走支付渠道；**后台调整/分配** 不涉及收款，需运营自行把握合规。
- 三种方式 **都不会** 自动扣余额；**都不会** 代替用户做「自动续费代扣」（仍须主动下单或人工操作）。
- 续期只延长 **订阅有效期**；若需把已用尽的 **日/周/月用量** 清零，另用「重置配额」。
