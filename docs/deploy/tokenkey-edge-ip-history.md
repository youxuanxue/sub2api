# TokenKey Edge Egress IP History & Replacement Runbook

This document tracks the egress (Elastic IP) lifecycle of TokenKey's AWS Stage0 edge gateways: which IPs are currently active, which have been retired (typically because an upstream API risk-blocked them — referred to here as "pollution"), and the validated procedure for replacing a polluted IP without breaking the CFN stack.

Companions:
- [`deploy/aws/stage0/edge-targets.json`](../../deploy/aws/stage0/edge-targets.json) — edge inventory; carries `drift_locked` + `drift_reason` per-edge.
- [`deploy/aws/stage0/edge-polluted-ips.json`](../../deploy/aws/stage0/edge-polluted-ips.json) — single source of truth for § 2 below.
- [`deploy/aws/cloudformation/stage0-edge-ec2.yaml`](../../deploy/aws/cloudformation/stage0-edge-ec2.yaml) — edge CFN template (ElasticIP + EIPAssoc carry `DeletionPolicy: Retain` and `UpdateReplacePolicy: Retain`).
- [`scripts/edge-ip-status.sh`](../../scripts/edge-ip-status.sh) — generator for the § 1 / § 2 tables; run after every replacement.

## 4. Replacement runbook (when a current IP is polluted)

Validated end-to-end against eu-west-2 / edge-uk1 on 2026-05-20.

> **Skill shortcut:** the entire replacement flow below is also packaged as the [`tokenkey-stage0-edge-ip-rotation`](../../.cursor/skills/tokenkey-stage0-edge-ip-rotation/SKILL.md) skill — Claude / Cursor agents can drive the procedure step-by-step from there. Use this prose runbook when reading on your own, and the skill when running it under an agent.

### Preconditions

- AWS CLI configured for the target region.
- SSM access — the edge instance's IAM profile must include `AmazonSSMManagedInstanceCore` (it does for `tokenkey-edge-*-stage0` stacks).
- Account-level EIP quota in the region (AWS default = 5). Replacement uses 1–4 candidate slots above the existing footprint; if the quota is tight, request an increase or re-allocate in smaller batches.
- The polluted edge is removed from any live routing pool (or you accept the cut-over window).

### Steps

1. **(Optional) Lower DNS TTL** on the edge's domain at the DNS provider (Porkbun today) to 60 s, one TTL cycle before the switch. Skipping this means up to ~10 min of stale external resolution after the switch.

2. **Allocate candidate EIPs** in the target region, but do not associate yet. One per available quota slot; 4 is enough in practice.
   ```bash
   aws ec2 allocate-address --region "$REGION" --domain vpc \
     --tag-specifications "ResourceType=elastic-ip,Tags=[{Key=Name,Value=<edge>-candidate-N},{Key=tokenkey:purpose,Value=ip-replacement-$(date -u +%F)}]"
   ```
   If any candidate matches an entry in § 2, immediately `release-address` and re-allocate.

3. **Launch a t4g.nano probe** in the same VPC/subnet/SG as the edge instance, reusing the edge's IAM profile (so SSM works out of the box). AMI = latest AL2023 ARM64. The probe is short-lived (< 30 min).
   ```bash
   EDGE_INSTANCE=$(aws cloudformation describe-stack-resources --region "$REGION" \
     --stack-name "$EDGE_STACK" \
     --query 'StackResources[?LogicalResourceId==`Instance`].PhysicalResourceId' --output text)
   EDGE_IAM_PROFILE=$(aws ec2 describe-instances --region "$REGION" --instance-ids "$EDGE_INSTANCE" \
     --query 'Reservations[0].Instances[0].IamInstanceProfile.Arn' --output text | awk -F/ '{print $NF}')
   EDGE_SUBNET=$(aws ec2 describe-instances --region "$REGION" --instance-ids "$EDGE_INSTANCE" \
     --query 'Reservations[0].Instances[0].SubnetId' --output text)
   EDGE_SG=$(aws ec2 describe-instances --region "$REGION" --instance-ids "$EDGE_INSTANCE" \
     --query 'Reservations[0].Instances[0].SecurityGroups[0].GroupId' --output text)
   AL2023_ARM_AMI=$(aws ec2 describe-images --region "$REGION" --owners amazon \
     --filters "Name=name,Values=al2023-ami-2023.*-arm64" "Name=state,Values=available" \
              "Name=architecture,Values=arm64" \
     --query 'sort_by(Images,&CreationDate)[-1].ImageId' --output text)

   aws ec2 run-instances --region "$REGION" \
     --image-id "$AL2023_ARM_AMI" --instance-type t4g.nano \
     --iam-instance-profile "Name=$EDGE_IAM_PROFILE" \
     --subnet-id "$EDGE_SUBNET" --security-group-ids "$EDGE_SG" \
     --associate-public-ip-address \
     --metadata-options "HttpTokens=required,HttpEndpoint=enabled" \
     --tag-specifications "ResourceType=instance,Tags=[{Key=Name,Value=<edge>-ip-probe-$(date -u +%F)}]"
   ```
   Wait for SSM `PingStatus=Online` (typically 60–120 s).

4. **Pollution detection.** The canonical pass signal is HTTP 401 / 400 with provider-shaped error JSON. The canonical fail signal is HTTP 403 with a Cloudflare challenge HTML body. For each candidate:
   - `aws ec2 associate-address --instance-id "$PROBE" --allocation-id "$CAND" --allow-reassociation`
   - Wait ~10 s for EIP propagation.
   - `aws ssm send-command` running:
     ```bash
     # Confirm outbound IP first
     test "$(curl -s --max-time 10 https://api.ipify.org)" = "$EXPECTED_IP" || exit 1

     # Then probe each upstream
     curl -sS --max-time 15 -o /tmp/_b -w '%{http_code}\n' \
       -H 'x-api-key: dummy' \
       -X POST -H 'content-type: application/json' \
       --data '{"model":"claude-sonnet-4-6","max_tokens":1,"messages":[{"role":"user","content":"hi"}]}' \
       https://api.anthropic.com/v1/messages

     curl -sS --max-time 15 -o /tmp/_b -w '%{http_code}\n' \
       -H 'authorization: Bearer dummy' \
       -X POST -H 'content-type: application/json' \
       --data '{"model":"gpt-4","messages":[{"role":"user","content":"hi"}]}' \
       https://api.openai.com/v1/chat/completions

     curl -sS --max-time 15 -o /tmp/_b -w '%{http_code}\n' \
       'https://generativelanguage.googleapis.com/v1beta/models?key=dummy'
     ```
   - Pass = all three application-layer 401 / 400. Any 403 + Cloudflare HTML in the body = polluted.

5. **Switch the live EIP**:
   ```bash
   aws ec2 associate-address --region "$REGION" \
     --instance-id "$EDGE_INSTANCE" --allocation-id "$CHOSEN_CAND" --allow-reassociation
   ```
   The old EIP is automatically `disassociate`d but **retained** in the account (no auto-release).

6. **Tag the new EIP** so future replacement runs (and `scripts/edge-ip-status.sh`) can see the change date:
   ```bash
   aws ec2 create-tags --region "$REGION" --resources "$CHOSEN_CAND" \
     --tags "Key=Name,Value=tokenkey-<edge>-eip" \
            "Key=tokenkey:status,Value=active" \
            "Key=tokenkey:replaced-on,Value=$(date -u +%F)" \
            "Key=tokenkey:replaces,Value=$OLD_IP"
   ```

7. **Update DNS** at the provider so the edge domain points at the new EIP.

8. **Verify from an independent observation point.** Do NOT rely on `curl` from a laptop — many local setups (VPN, Docker, mitmproxy, Charles, transparent proxies) silently hijack outbound to public IPs even with `--resolve`, producing `remote_ip=127.0.0.1`. The simplest reliable verifier is the probe instance, still running from step 3:
   ```bash
   aws ssm send-command --region "$REGION" --instance-ids "$PROBE" \
     --document-name AWS-RunShellScript \
     --parameters 'commands=["curl -sS --max-time 12 -o /tmp/r -w \"http=%{http_code} ip=%{remote_ip}\\n\" https://<edge-domain>/robots.txt"]'
   ```
   Expect `http=200 ip=<NEW_EIP>` and a sensible response body.

9. **Lock the edge against accidental deploy** (mechanical guard for § 5 not being run yet):
   ```bash
   jq --arg edge "<edge>" --arg ip_old "$OLD_IP" --arg ip_new "$CHOSEN_IP" \
      '.targets[$edge].drift_locked = true |
       .targets[$edge].drift_reason  = "EIP replaced \(now | strftime("%Y-%m-%d")) outside CFN (\($ip_old) polluted → \($ip_new)); recovery in docs/deploy/tokenkey-edge-ip-history.md § 5 required before next deploy."' \
      deploy/aws/stage0/edge-targets.json > /tmp/m && mv /tmp/m deploy/aws/stage0/edge-targets.json
   ```

10. **Cleanup**:
    - `aws ec2 terminate-instances --instance-ids "$PROBE"`.
    - `aws ec2 release-address` every unused candidate EIP.
    - `aws ec2 release-address` the polluted EIP itself, then add it to `deploy/aws/stage0/edge-polluted-ips.json` (see § 2 example shape). Re-run `scripts/edge-ip-status.sh --markdown` and paste the new tables back into § 1 / § 2 of this doc.

11. **CFN drift recovery** — every EIP replacement done outside the template introduces drift. Run § 5 before the next `deploy-edge-stage0.yml` against this edge.

## 5. CFN drift recovery runbook

The Stage0 edge template (`deploy/aws/cloudformation/stage0-edge-ec2.yaml`) creates both `ElasticIP` and `EIPAssoc` via CFN. The replacement procedure in § 4 swaps them at the AWS layer — CFN's view stays frozen on the old physical IDs.

### Retain policies are already in place

The accompanying template change introduces:

```yaml
  ElasticIP:
    Type: AWS::EC2::EIP
    DependsOn: IGWAttach
    DeletionPolicy: Retain
    UpdateReplacePolicy: Retain
    Properties:
      Domain: vpc
      Tags: [...]

  EIPAssoc:
    Type: AWS::EC2::EIPAssociation
    DeletionPolicy: Retain
    UpdateReplacePolicy: Retain
    Properties:
      AllocationId: !GetAtt ElasticIP.AllocationId
      InstanceId: !Ref Instance
```

With these policies, future EIP replacements (§ 4) cannot accidentally destroy a manually swapped EIP on the next stack update. The IMPORT phase below is still required to realign CFN's view with reality.

### Resource-identifier reference (verify before running)

CloudFormation IMPORT requires a specific `ResourceIdentifier` key per resource type. As of 2026-05-20, per the [AWS::EC2::EIP](https://docs.aws.amazon.com/AWSCloudFormation/latest/UserGuide/aws-resource-ec2-eip.html) and [AWS::EC2::EIPAssociation](https://docs.aws.amazon.com/AWSCloudFormation/latest/UserGuide/aws-properties-ec2-eip-association.html) documentation:

| Resource type | IMPORT `ResourceIdentifier` key | Identifier format |
| --- | --- | --- |
| `AWS::EC2::EIP` | `AllocationId` | `eipalloc-…` |
| `AWS::EC2::EIPAssociation` | `Id` | `eipassoc-…` (the association id) |

> Before executing the recovery against a production stack, run `aws cloudformation create-change-set --change-set-type IMPORT --no-execute` (Phase 2 step 4 below) on a dev / staging stack first and inspect the change-set diff. If AWS rejects the identifier shape (very unlikely but possible if the IMPORT spec changes), update this table and the recovery procedure before continuing.

### Recovery procedure (one-shot, per drifted edge)

Two-phase: (1) apply the Retain template so the EIP can survive subsequent stack updates, (2) reseat the live EIP back into the stack via IMPORT.

> **Phase 1 may fail outright** if CFN rejects the drifted `ElasticIP` resource (physical id points at a released EIP). That is acceptable — skip directly to Phase 2 (detach + reimport).

#### Phase 1 — apply Retain policy without changing parameters

The replacement runbook leaves `drift_locked: true` on the edge, so `deploy-edge-stage0.yml` is blocked. Manually apply the template change with all parameters carried over from the current stack:

```bash
REGION=eu-west-2
STACK=tokenkey-edge-uk1-stage0
PARAMS=$(aws cloudformation describe-stacks --region "$REGION" --stack-name "$STACK" \
  --query 'Stacks[0].Parameters[].ParameterKey' --output text | \
  tr '\t' '\n' | awk 'NF{print "ParameterKey="$0",UsePreviousValue=true"}')

aws cloudformation update-stack --region "$REGION" --stack-name "$STACK" \
  --template-body file://deploy/aws/cloudformation/stage0-edge-ec2.yaml \
  --capabilities CAPABILITY_IAM CAPABILITY_NAMED_IAM \
  --parameters $PARAMS
```

- If `update-stack` completes: drift still exists at the physical-id level; continue to Phase 2.
- If `update-stack` fails (`ROLLBACK_COMPLETE` or `cannot find resource …`): cancel any rollback and continue to Phase 2.

#### Phase 2 — detach + reimport

Goal: drop `ElasticIP` + `EIPAssoc` from the stack (Retain protects the real resources), then reimport them at their current physical IDs.

1. **Edit the template locally — DO NOT commit this temporary change.** Comment out (do not delete) the `ElasticIP` and `EIPAssoc` blocks in `deploy/aws/cloudformation/stage0-edge-ec2.yaml`. Save the file as `/tmp/stage0-edge-ec2.detach.yaml`.

2. **Apply the detached template:**
   ```bash
   aws cloudformation update-stack --region "$REGION" --stack-name "$STACK" \
     --template-body file:///tmp/stage0-edge-ec2.detach.yaml \
     --capabilities CAPABILITY_IAM CAPABILITY_NAMED_IAM \
     --parameters $PARAMS
   aws cloudformation wait stack-update-complete --region "$REGION" --stack-name "$STACK"
   ```
   The two resources leave the stack; the live EIP and association on AWS are untouched (because of Retain).

3. **Prepare the IMPORT change-set inputs.** Find current physical IDs:
   ```bash
   NEW_ALLOC=$(aws ec2 describe-addresses --region "$REGION" \
     --filters "Name=tag:Name,Values=tokenkey-uk1-eip" \
     --query 'Addresses[0].AllocationId' --output text)
   NEW_ASSOC=$(aws ec2 describe-addresses --region "$REGION" --allocation-ids "$NEW_ALLOC" \
     --query 'Addresses[0].AssociationId' --output text)
   ```
   Write `resources-to-import.json`:
   ```json
   [
     { "ResourceType": "AWS::EC2::EIP",          "LogicalResourceId": "ElasticIP", "ResourceIdentifier": { "AllocationId": "REPLACE_ME" } },
     { "ResourceType": "AWS::EC2::EIPAssociation","LogicalResourceId": "EIPAssoc","ResourceIdentifier": { "Id": "REPLACE_ME" } }
   ]
   ```
   substituting `$NEW_ALLOC` and `$NEW_ASSOC`.

4. **Create the IMPORT change-set against the ORIGINAL committed template** (the one with `ElasticIP` + `EIPAssoc` resources defined, i.e. the file in repo, not the detached one):
   ```bash
   aws cloudformation create-change-set --region "$REGION" \
     --stack-name "$STACK" \
     --change-set-name eip-import-$(date -u +%F) \
     --change-set-type IMPORT \
     --resources-to-import file://resources-to-import.json \
     --template-body file://deploy/aws/cloudformation/stage0-edge-ec2.yaml \
     --capabilities CAPABILITY_IAM CAPABILITY_NAMED_IAM
   ```

5. **Review the change-set carefully:**
   ```bash
   aws cloudformation describe-change-set --region "$REGION" \
     --stack-name "$STACK" \
     --change-set-name eip-import-$(date -u +%F) \
     --query 'Changes[].{Action:ResourceChange.Action,Type:ResourceChange.ResourceType,Logical:ResourceChange.LogicalResourceId,Physical:ResourceChange.PhysicalResourceId}' \
     --output table
   ```
   Confirm: only `ElasticIP` and `EIPAssoc` are touched (action = `Import`), no other resource is being replaced or deleted. If any unexpected resource appears, stop and re-evaluate.

6. **Execute:**
   ```bash
   aws cloudformation execute-change-set --region "$REGION" \
     --stack-name "$STACK" \
     --change-set-name eip-import-$(date -u +%F)
   aws cloudformation wait stack-update-complete --region "$REGION" --stack-name "$STACK"
   ```

7. **Verify drift cleared:**
   ```bash
   DRIFT_ID=$(aws cloudformation detect-stack-drift --region "$REGION" --stack-name "$STACK" \
     --query 'StackDriftDetectionId' --output text)
   sleep 30
   aws cloudformation describe-stack-drift-detection-status --region "$REGION" \
     --stack-drift-detection-id "$DRIFT_ID" \
     --query '{State:DetectionStatus,Drift:StackDriftStatus}' --output table
   # Expect: DetectionStatus=DETECTION_COMPLETE, StackDriftStatus=IN_SYNC
   aws cloudformation describe-stack-resources --region "$REGION" --stack-name "$STACK" \
     --query 'StackResources[?ResourceType==`AWS::EC2::EIP` || ResourceType==`AWS::EC2::EIPAssociation`].{Logical:LogicalResourceId,Type:ResourceType,Physical:PhysicalResourceId}' \
     --output table
   # Expect ElasticIP physical-id = the new EIP (35.177.124.150 for uk1),
   #        EIPAssoc physical-id = the live eipassoc-… created by step 5 of § 4.
   ```

8. **Clear `drift_locked`** in `deploy/aws/stage0/edge-targets.json` for the recovered edge, commit, push, open PR. The next `deploy-edge-stage0.yml` against this edge is now safe.

9. **Smoke** via the probe pattern in § 4 step 8 (or via `deploy-edge-stage0.yml` smoke-only mode) before reactivating that edge in the routing pool.

> **Operate Phase 2 as a deliberate human action, not from automation.** It rewires CFN ownership of production resources. If the change-set diff in step 5 shows anything beyond `ElasticIP` / `EIPAssoc`, stop and re-evaluate.

## 6. Decommissioning an edge stack

The Retain policy added by this PR means `aws cloudformation delete-stack` will NOT release the EIP or remove the association. If you really want to retire an edge:

```bash
REGION=<region>
STACK=tokenkey-edge-<id>-stage0

# 1. Find and release the EIP first.
EIP_ALLOC=$(aws cloudformation describe-stack-resources --region "$REGION" --stack-name "$STACK" \
  --query 'StackResources[?ResourceType==`AWS::EC2::EIP`].PhysicalResourceId | [0]' --output text)
# For a normal (non-drifted) stack, PhysicalResourceId is the public IPv4 address.
# Resolve to AllocationId via describe-addresses:
EIP_ALLOC_ID=$(aws ec2 describe-addresses --region "$REGION" --public-ips "$EIP_ALLOC" \
  --query 'Addresses[0].AllocationId' --output text)
aws ec2 release-address --region "$REGION" --allocation-id "$EIP_ALLOC_ID"

# 2. Delete the stack (Retain on EIP/EIPAssoc no longer matters — both are already gone).
aws cloudformation delete-stack --region "$REGION" --stack-name "$STACK"
aws cloudformation wait stack-delete-complete --region "$REGION" --stack-name "$STACK"

# 3. Remove the edge entry from deploy/aws/stage0/edge-targets.json in a PR.
# 4. Re-run scripts/edge-ip-status.sh and refresh § 1 of this doc.
```

If the edge is drift-locked (manually replaced EIP), the EIP's `PhysicalResourceId` in CFN is stale; use `aws ec2 describe-addresses --filters Name=tag:Name,Values=tokenkey-<id>-eip` to find the live AllocationId before releasing.

Other Retain-able resources on the edge stack (alarms, parameter store entries) are reaped automatically on `delete-stack` because they do not carry `DeletionPolicy: Retain`.
