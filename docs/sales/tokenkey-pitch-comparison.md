# TokenKey 销售说辞 · 对比版

> 适用场景：客户需要逐条对比、对技术细节有兴趣、采购决策需要"证据清单"。
> 全部论点基于代码事实可现场验证：9 个 deployable 边缘节点、原生 `/v1/messages` 全保真、实测可服务模型目录（最近实测 2026-06-05）。

---

## 一、两句话（远超国内同行）

> **第一句：「同行卖的是号，我们卖的是基础设施。」**
> 国内中转站多是单机 + 一批账号，账号死了服务就死。TokenKey 是 **9 个全球边缘节点 + 生产中枢的两层架构**——任一账号、任一节点、任一出口 IP 被风控，流量秒级切到下一个，客户端无感知。

> **第二句：「同行靠人肉救火，我们一切代码化、自动化。」**
> 调度、限流、容灾、发布、告警、上游同步全部自动化且版本化：每小时离机备份、一键回滚、账号异常实时告警自愈、每天自动合并上游更新——**没有一个环节靠某个人半夜手工处理。**

## 二、稳定性证明（每条都是可验证的工程事实）

| 维度 | 国内同行典型做法 | TokenKey |
|---|---|---|
| **架构** | 单机 + 一批账号 | 生产中枢 + 9 个全球边缘节点，prod→edge 镜像中继，跨节点自动 failover |
| **账号存活** | 裸调用、指纹裸奔、频繁封号 | **TLS 指纹 + HTTP 头与真实 Claude Code 逐字节对齐**（JA3、UA、beta header 全套），专门的抓包→diff→修复流水线，Claude Code 每次发版后 24h 内重新校准 |
| **限流处理** | 429 直接抛给客户 | 账号级 5h/7d 用量窗口感知调度、冷却自动恢复、空池快速失败不挂死连接 |
| **故障恢复** | 群里喊"修复中" | 数据库每小时离机备份到 S3（RPO≤1h）、灾备手册代码化、换机零数据丢失 |
| **监控告警** | 客户先发现问题 | 账号失效秒级飞书告警、每节点健康度自动巡检、错误分相归因（区分客户端错误 vs 上游故障） |
| **可服务模型** | 列表靠抄 | **逐模型真实请求实测**，只列返回 200 的，定期自动刷新（最近 2026-06-05） |

## 三、与 OpenRouter 对比

| | OpenRouter | TokenKey |
|---|---|---|
| **Claude Code 体验** | 走 OpenAI 兼容协议，**丢失原生特性** | **原生 `/v1/messages` 全保真**：1M context、extended thinking、prompt caching、count_tokens 全部透传，换 base_url 即用。客户可现场验证：把 `ANTHROPIC_BASE_URL` 指向我们跑一个 **extended thinking + 1M context** 的任务——国内绝大多数中转跑不通这个组合 |
| **限流韧性** | 单账号配额打满即 429 | 多账号多节点池化，窗口感知调度自动腾挪 |
| **模型广度** | 数百模型（这是它的强项，承认） | 聚焦六平台精品：Claude 全系（含 opus-4-8）、GPT-5 全系/Codex、Gemini/Vertex、Kiro、任意 OpenAI 兼容上游 |

> **对 OpenRouter 的一句话定位：** OpenRouter 是"模型超市"，适合要广度的人；TokenKey 是"Claude/GPT 重度生产用户的专线"——**同样的 Claude Code 工作流，原生体验不打折，且为封号和限流做了 OpenRouter 不需要也不会做的全套工程对抗。**
