# v1.5.0 NewAPI 添加账号模态框两个真实 bug 的深度诊断与修复方案

> 用户复现：登录 https://api.tokenkey.dev/admin/accounts → 「添加账号」。
>
> - 截图 1：默认状态（`form.platform = 'anthropic'`）下点击 **NewAPI** segment → 模态框只剩 备注 / 平台 / 临时不可调度 / 代理 / 并发因子 / 优先级 / 账号计费倍率 / 过期时间 / 过期自动暂停调度 / 分组。**没有** 类型(channel_type) / Base URL / API Key / 模型 / 获取模型列表。
> - 截图 2：先点 **OpenAI → API Key**，再点 **NewAPI** → 模态框出现 Base URL / API Key / 模型限制白名单（37 个 OpenAI 模型）。但 **没有** 类型(channel_type) 选择，也 **没有** 「获取模型列表」按钮。Base URL 占位符显示 `https://api.anthropic.com`（错误占位符）。
> - 对比截图 3（new-api 原生 / GLM-style channel 创建表单）：类型下拉 + 密钥 + API地址 + 模型多选 + **「获取模型列表」按钮** + 模型重定向（可视化/手动 JSON）。
>
> 这两个现象不是孤立的 UX 问题，是**两个真实 bug 叠加在 v1.5.0 同一条调用链上**。本文给出根因、影响、修复方案与防回归门禁。

```ts

  platform: 'anthropic' as AccountPlatform,
  type: 'oauth' as AccountType,
  ...

// CreateAccountModal.vue:3089-3090
const accountCategory = ref<'oauth-based' | 'apikey' | 'bedrock'>('oauth-based')

// CreateAccountModal.vue:3444-3464
watch([accountCategory, addMethod, antigravityAccountType], ([category, method, agType]) => {
  if (form.platform === 'antigravity' && agType === 'upstream') { form.type = 'apikey'; return }
  if (form.platform === 'anthropic' && category === 'bedrock') { form.type = 'bedrock'; return }
  if (category === 'oauth-based') {
    form.type = method as AccountType   // 'oauth' | 'setup-token'
  } else {
    form.type = 'apikey'
  }
}, { immediate: true })
```

**关键事实**：watcher 在初始化时立即执行一次 → 当模态框第一次打开（`form.platform='anthropic'`、`accountCategory='oauth-based'`、`method='oauth'`）→ `form.type='oauth'`。点击 NewAPI segment 仅修改 `form.platform = 'newapi'`，**不触发** `[accountCategory, addMethod, antigravityAccountType]` 任何依赖变化 → `form.type` 仍为 `'oauth'`。

切换 OpenAI → API Key 时，按钮 `@click="accountCategory = 'apikey'"` 触发 watcher → `form.type = 'apikey'`。再点 NewAPI 仅改 `form.platform`，`form.type='apikey'` 残留。

## 2. Bug A 深度诊断（截图 1：NewAPI 整段不渲染）

### 2.1 三种可能根因

#### 2.1.1 (A1) 部署镜像的 frontend dist 是 PR #19 之前的旧产物

最可信的解释。论据链：

1. v1.5.0 的 git tag (commit `7cad11d4`) 包含 PR #19（commit `3a43a109`）和 PR #29（commit `92dcc230`）。
2. PR #19 第一次把 `<AccountNewApiPlatformFields>` 接到 CreateAccountModal.vue:729（`git show v1.5.0:frontend/src/components/account/CreateAccountModal.vue` 已包含完整 wiring）。
3. v1.5.0 release pipeline (`.github/workflows/release.yml`) 的 `build-frontend` job 使用 `actions/checkout@v6` + `pnpm install --frozen-lockfile && pnpm run build`，应当编译 v1.5.0 commit 的源码。
4. 在干净的 workspace 复测：`cd frontend && pnpm build` 成功，`backend/internal/web/dist/assets/AccountsView-*.js` 包含 `AccountNewApiPlatformFields` / `newapiChannelType` / `listChannelTypes` 字符串（已 grep 验证）。
5. 但 `aws prod` 显示**完全没有**这些字段——只可能是部署的 image 实际不是从 v1.5.0 的 release pipeline 来的，而是：
   - 用了旧的本地 `docker build` 镜像并打了 `:1.5.0` 标签（manual override）
   - 用了 `:main` 浮动 tag，且 `:main` 在 PR #19 之前最后一次构建（`deploy/aws/stage0/.env.example` 默认就是 `TOKENKEY_IMAGE=ghcr.io/<owner>/sub2api:main`）
   - SSM/容器在拉取时使用了缓存层，没有强制 pull 最新 sha
6. 复核办法（运维侧）：
   ```bash
   ssh ec2-user@<prod-ec2>
   docker inspect $(docker ps -q --filter ancestor=ghcr.io/<owner>/sub2api) \
     | jq '.[0].Config.Labels."org.opencontainers.image.revision"'
   ```
   把这条 commit SHA 与 `git rev-list v1.5.0` 对照。如果 SHA 不在 v1.5.0 链上 → 确认 (A1)。

#### 2.1.2 (A2) 运行时 vue-i18n 缺 key 导致组件 setup 抛错

`AccountNewApiPlatformFields.vue` 的 setup 块在创建时同步调用 `useI18n()` 并不立即解析 key（template 渲染时才解析）。但如果 `<Select :placeholder="t('admin.accounts.newApiPlatform.channelTypePlaceholder')">` 在 `t()` 调用层抛 throw（例如 missingWarn=throw 模式），会让组件子树渲染失败。复核：

```bash
grep -rE "newApiPlatform\b" frontend/src/i18n/locales/{zh,en}.ts | head -10
```

已确认 zh.ts / en.ts 都有 `newApiPlatform.*` key；vue-i18n 默认 `missingWarn=true`、`fallbackWarn=true`、**不**抛错。所以 (A2) 排除。

#### 2.1.3 (A3) Vue 模板编译 bug 让 `v-if` 链断裂

不可能。我们已用 `pnpm typecheck` 和 `pnpm build` 验证编译通过，未报告任何 v-if 错误。

### 2.2 结论

**Bug A 几乎确定是 (A1) 部署镜像与 v1.5.0 源码不一致**——具体是部署链路在哪一环用了旧镜像，需要运维一条命令复核（§2.1.1 步骤 6）。如果 (A1) 被运维证伪（部署镜像 SHA 确实在 v1.5.0 链上），那么 fallback 是 (A4)：**特定浏览器扩展或 CSP 把内联脚本拦截**——但此时 `临时不可调度` 等其他子段同样依赖 i18n 与 reactive watch，不会单独保留它们而独丢 newapi 段。所以工程概率 ~99% 是 (A1)。

### 2.3 需要的 evidence

```
1. 运维提供 docker inspect 的 image revision label
2. 运维提供 GHCR pull 时间戳（与 v1.5.0 release timestamp 对照）
3. 浏览器 DevTools Network 中 AccountsView-*.js 的实际 hash（与本地 pnpm build 出的 hash 对照）
```

## 3. Bug B 深度诊断（"获取模型列表" 按钮永远不存在）

### 3.1 根因

PR #19 设计文档 `docs/approved/admin-ui-newapi-platform-end-to-end.md` § 1.6 中明确把 `useTkAccountNewApiPlatform` composable + `upstreamFetchConfig` 作为 stage-2/3 的能力交付项。代码层面：

- `useTkAccountNewApiPlatform.ts` ✅ 已实现（含 `fetchUpstreamModels` + `upstreamFetchConfig` + `bootstrapNewapiCatalog` + `isNewApiUpstreamFetchableChannelType` 守卫）
- `AccountNewApiPlatformFields.vue` ❌ 未渲染按钮、未接收 fetch 配置 prop / slot
- `CreateAccountModal.vue` ❌ 未 import `useTkAccountNewApiPlatform`
- `EditAccountModal.vue` ❌ 未 import `useTkAccountNewApiPlatform`

实际生效的是一条**简化版直挂**：`CreateAccountModal.vue:3415-3424` 直接调 `listChannelTypes()` 加载下拉选项，但没有"按 channel_type 触发 fetchUpstreamModels 把模型注入白名单"这一动作。

### 3.2 影响

- **用户必须自己手填**所有上游模型 ID（截图 2 中 OpenAI 那 37 个模型实际上是 model whitelist 的模板预填，不是 newapi 上游模型；用户如果在 newapi 分组下用这份白名单实际会调度失败）。
- 开发者在 PR #29 提交时认为该功能已 ship（因为后端 + composable + i18n 全都到位），但 UI 端没有实际暴露按钮——这是典型的"有零件、未组装"漏洞。

### 3.3 与 new-api 原生形态的对照（截图 3）

new-api 原生 `创建新的渠道` 表单（来自 `/new-api/web/src/.../EditChannel.jsx`）有：

| 控件 | TokenKey 状态 |
|---|---|
| 类型（channel_type 下拉） | ✅（`AccountNewApiPlatformFields` 已有） |
| 名称 | ✅ |
| 密钥（api_key） | ✅ |
| API地址（base_url） | ✅ |
| 模型多选 | ❌（被外层 Model Restriction 占据，但条件不对） |
| **获取模型列表 按钮** | ❌ **本文 bug B** |
| **填入相关模型 按钮**（用 channel_type 默认列表填充） | ⚠️ 部分实现：`useNewApiChannelTypeModels` composable 写好了；当 channel_type 变化时 `useTkAccountNewApiPlatform.ts:119-139` 会自动同步进 allowedModels；但因为 composable 未被引入 CreateAccountModal，行为同样未触达 |
| 自定义模型名称 | ❌（没有手填新模型 ID 的入口） |
| 分组 | ✅（外层通用字段） |
| 模型重定向（model_mapping） | ✅（`AccountNewApiPlatformFields` 第 4 个字段：JSON-object textarea） |

一句话总结：TokenKey 的 newapi 子组件**字段集是齐的**，但**自动填充与上游 fetch 的两个动作按钮缺位**。

## 4. Bug C 深度诊断（截图 2 中 NewAPI 错误共享 OpenAI/API Key 段）

### 4.1 触发链

1. 用户先点 OpenAI segment → `form.platform = 'openai'`，初始 `accountCategory = 'oauth-based'`
2. 用户再点 OpenAI 下的 **API Key** card → `accountCategory = 'apikey'` → watcher 触发 `form.type = 'apikey'`
3. 用户再点 NewAPI segment → `form.platform = 'newapi'`（**`form.type` 没被重置，仍 = 'apikey'**）
4. 模板渲染：
   - 行 729 (`form.platform === 'newapi'`) ✅ 渲染 → AccountNewApiPlatformFields（channel_type / base_url / api_key / model_mapping / ...）
   - 行 890 (`form.type === 'apikey' && form.platform !== 'antigravity'`) ✅ **也渲染** → 第二份 base_url/api_key/Model Restriction（来自 OpenAI 上下文）
5. 截图 2 显示的就是行 890 这部分（**因为屏幕只截了一段，行 729 的 AccountNewApiPlatformFields 实际可能在更上面，或被 bug A 的同一根因遮蔽**）。

### 4.2 行 890 的占位符 ternary 也漏判 newapi

```vue
:placeholder="
  form.platform === 'openai'  ? 'https://api.openai.com'
  : form.platform === 'gemini' ? 'https://generativelanguage.googleapis.com'
                                : 'https://api.anthropic.com'  // ← newapi 落到这里
"
```

`form.platform === 'newapi'` 落到兜底 anthropic 占位符——这正是截图 2 中"Base URL: https://api.anthropic.com"的来源，**与上游协议完全不符**。

### 4.3 影响（按风险）

| 风险 | 表现 |
|---|---|
| 误导 | 用户看到"https://api.anthropic.com"会错以为是 newapi 默认值 |
| 数据污染 | 如果用户没注意到上方 AccountNewApiPlatformFields 已正确填了 base_url，又在下方"被误导的 base_url 输入框"里填了别的值，提交 payload 会以**下方**值覆盖（取决于 `submit()` 取哪份 ref） |
| Model Restriction 段误预填 | OpenAI 模型白名单（gpt-3.5-turbo, gpt-4, gpt-5.x …）在 newapi 分组下完全无效——newapi 账号的白名单应该来自该 channel_type 的 default model list 或 fetch upstream models |
| 配置漂移 | 提交后写入的账号实际有两套字段：credentials.{api_key, base_url, channel_type, model_mapping, status_code_mapping, openai_organization}（来自 AccountNewApiPlatformFields）+ allowed_models（来自 OpenAI 模板）。后端能容忍但调度器/bridge 行为模糊 |

### 4.4 后端 submit 路径验证

`CreateAccountModal.vue:4118-4180`（newapi 分支提交）：

```ts
if (form.platform === 'newapi') {
  if (!newapiChannelType.value || newapiChannelType.value <= 0) { /* 报错 */ return }
  const baseUrl = newapiBaseUrl.value.trim() || newapiSelectedBaseUrl.value

    platform: 'newapi',

```

**确认数据污染**：`allowedModels.value` 会被 OpenAI 模板预填的 `[...claudeModels]`（resetForm:3881）或 `[...getModelsByPlatform('openai')]`（show watcher:3414）污染——**只要用户没主动清空白名单，提交后 newapi 账号会带着 OpenAI 模型白名单上线**，调度时 `IsModelSupported` 检查会基本必拒。
